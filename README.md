# FinPick (FinPilot) 개발 일지

FinPick은 **투자 성향 기반 맞춤형 주식 종목 추천 + 금융 학습 서비스**입니다.
기능 단위(F1xx)로 나누어 구현하며, 각 섹션은 해당 기능의 구현 내용·모델 설계·
어려웠던 점·느낀점을 담은 개발 기록입니다.

**공통 기술 스택**: Django REST Framework(백엔드) · Vue 3 + Pinia(프론트엔드) · JWT 인증 · SQLite(개발)

| 기능 | 내용 | 상태 |
|---|---|---|
| F100 | 회원관리 (인증) | ✅ 완료 |
| F200 | 투자성향 (선호 입력) | ✅ 완료 |
| F400 | 종목 조회 (목록·검색·상세) | ✅ 완료 |
| F300 | AI 종목 추천 | ✅ 완료 |

---

## F100 회원관리 기능 구현

FinPick(투자 성향 기반 맞춤형 주식 종목 추천 서비스)의 첫 기능인 **F100 회원관리**를
구현한 기록입니다. 회원가입, 로그인, 토큰 갱신, 로그아웃, 회원정보 조회·수정까지의
인증 흐름을 Django REST Framework와 Vue 3로 구현했습니다.

**기술 스택**
- 백엔드: Django, Django REST Framework, `djangorestframework-simplejwt`
- 프론트엔드: Vue 3, Vue Router, Pinia, Axios
- 인증: JWT (Access + Refresh Token)
- DB: SQLite (개발)

---

### 구현 내용

#### 백엔드 API

| 기능 | Method | Endpoint |
|---|---|---|
| 아이디 중복확인 (F102) | GET | `/api/v1/accounts/check-username/?username=` |
| 회원가입 (F103) | POST | `/api/v1/accounts/signup/` |
| 로그인 (F105) | POST | `/api/v1/accounts/login/` |
| 토큰 갱신 (F106) | POST | `/api/v1/accounts/token/refresh/` |
| 로그아웃 (F107) | POST | `/api/v1/accounts/logout/` |
| 회원정보 조회·수정 (F108·F109) | GET/PATCH | `/api/v1/accounts/me/` |

- **회원가입**: 아이디 중복(대소문자 무관)·비밀번호 일치·비밀번호 정책·이메일 형식을
  검증하고, 비밀번호는 해시로 저장한다. 가입 성공 시 JWT를 발급해 바로 로그인 상태가 된다.
- **로그인 / 토큰 갱신**: 로그인 시 Access·Refresh 토큰을 발급하고, Access 만료 시
  Refresh로 재발급한다. 로그인 실패 메시지는 계정 존재 여부를 노출하지 않는 공통 오류로 처리한다.
- **로그아웃**: Refresh 토큰을 블랙리스트에 등록해 로그아웃 후 재사용을 차단한다.
- **회원정보 수정**: 비밀번호 변경 시 현재 비밀번호를 확인하고, 이메일은 본인을 제외한
  중복을 검증한다.

#### 프론트엔드 화면

- **회원가입 / 로그인 / 마이페이지** 화면을 다크 + 골드 테마로 구현
- 로그인 상태에 따라 헤더가 `로그인·회원가입` ↔ `마이페이지·로그아웃`으로 전환
- Axios 인터셉터로 모든 요청에 토큰을 자동 첨부하고, 401 발생 시 Refresh 토큰으로
  자동 재발급 후 재시도
- 라우터 가드로 비로그인 사용자의 마이페이지 접근을 차단

### 모델 설계

회원 정보는 Django 기본 `auth.User` 대신 **커스텀 User 모델**(`accounts.User`)로 설계했다.

```python
class User(AbstractUser):
    email = models.EmailField("email address", unique=True)   # 이메일 중복 불가
    birth_date = models.DateField("생년월일", null=True, blank=True)

    @property
    def has_investment_profile(self) -> bool:
        # 투자성향(F200) 등록 여부 — 별도 모델과의 관계로 판단
        return hasattr(self, "investment_profile")
```

**설계 의도**
- `AbstractUser`를 상속해 username·password·date_joined 등 기본 인증 필드를 그대로 활용하면서,
  서비스에 필요한 `birth_date`(생년월일)와 `email` 유니크 제약만 추가했다.
- 투자성향 같은 부가 정보는 User에 직접 넣지 않고 별도 모델(F200, `InvestmentProfile`)과의
  1:1 관계로 분리해, User 모델은 인증 책임만 갖도록 했다.
- 프로젝트 초기에 커스텀 User로 전환해 두면 이후 필드 확장이 자유롭다.

---

### 어려웠던 점과 해결 방법

#### 1. 이미 마이그레이션된 기본 User 모델
F100 구현 직전, 이미 Django 기본 `auth.User`가 마이그레이션되어 있었다.
커스텀 User 모델로 바꾸려면 충돌이 발생하는 상황이었다.
→ 사용자가 0명인 개발 초기였으므로 **개발 DB를 초기화하고 커스텀 User 기준으로
다시 마이그레이션**했다. 초기에 결정한 덕분에 비용 없이 전환할 수 있었다.

#### 2. 로그아웃 후에도 토큰이 살아있는 문제
JWT는 기본적으로 서버에 상태를 저장하지 않아, 로그아웃해도 발급된 토큰은
만료 전까지 유효하다.
→ **`token_blacklist` 앱을 적용해 Refresh 토큰을 블랙리스트에 등록**하도록 했다.
로그아웃 후 해당 토큰으로 갱신을 시도하면 차단되는 것을 테스트로 확인했다.

#### 3. 화면 전환 후 인증 상태가 끊기는 문제
SPA에서 새로고침이나 페이지 이동 시 토큰을 잃지 않아야 했다.
→ 토큰을 **localStorage에 저장**하고 Axios 요청 인터셉터에서 자동으로 첨부하도록 했다.
Access 만료 시에는 응답 인터셉터가 Refresh로 한 번 갱신한 뒤 원래 요청을 재시도한다.

---

### 느낀점

- **인증은 "성공 경로"보다 "실패·경계 경로"가 핵심**이었다. 로그인 성공보다 잘못된
  비밀번호, 만료된 토큰, 로그아웃 후 재사용 같은 경우를 어떻게 처리하느냐가 더 중요했다.
- **초반 결정의 무게를 체감했다.** User 모델처럼 나중에 바꾸기 어려운 선택은,
  사용자가 없는 초기에 미리 정해두는 것이 비용을 크게 줄여준다는 걸 직접 겪었다.
- **명세서를 코드 ID와 1:1로 연결**(F102, F103 …)해두니, 무엇을 구현했고 무엇이 남았는지
  추적하기가 훨씬 수월했다. 다음 기능부터도 이 방식을 유지할 생각이다.
- 작은 도구 문제(PDF 인코딩) 하나로도 작업이 멈출 수 있다는 점에서, 막혔을 때
  **방법을 바꿔보는 판단**의 중요성을 다시 느꼈다.

## F200 투자성향 기능 구현

**F200 투자성향**은 사용자가 자신의 투자 선호 — 투자 가능 자산·위험 선호도·투자 기간·
투자 목적·관심 산업 — 를 직접 입력하고 조회·수정·삭제하는 기능입니다. 이 데이터는 이후
**F300 맞춤 추천의 입력**이 됩니다. 명세서 v2.0 §5.2의 "선호 직접 입력" 방식을 따랐고,
점수형 퀴즈가 아니라 **CRUD**로 구현했습니다.

**핵심 결정**: 점수·가중치 산정은 투자성향이 아니라 추천(F300 §5.3)의 책임으로 보고
F200에서는 다루지 않는다.

---

### 구현 내용

#### 백엔드 API (전부 인증 필요)

| 기능 | Method | Endpoint |
|---|---|---|
| 설문 선택지 조회 (F201) | GET | `/api/v1/investment-profile/options/` |
| 투자성향 조회 (F208) | GET | `/api/v1/investment-profile/` |
| 투자성향 저장 (F207, upsert) | POST | `/api/v1/investment-profile/` |
| 투자성향 수정 (F209) | PATCH | `/api/v1/investment-profile/` |
| 투자성향 삭제 (F210, soft) | DELETE | `/api/v1/investment-profile/` |

- **설문 선택지(F201)**: 위험·기간·목적·자산구간 enum과 활성 관심 산업 목록을 한 번에 내려,
  프론트가 선택지를 하드코딩하지 않고 서버 기준으로 그리도록 했다.
- **저장(F207, upsert)**: 사용자당 활성 프로필 1건만 유지한다. `update_or_create`로 신규
  등록과 수정을 공통 처리하고, 본문 저장과 관심 산업(M2M) 저장을 `transaction.atomic`으로
  묶어 **부분 저장 실패 시 전체 롤백**한다. 신규는 201, 수정은 200으로 응답.
- **조회·수정·삭제**: 미등록 상태에서의 조회·수정·삭제는 모두 `404 RESOURCE_NOT_FOUND`로
  통일했다. 삭제(F210)는 행을 지우지 않고 `is_active=False`로 비활성화하는 **soft delete**.
- **검증**: 관심 산업은 활성 산업만 남기고 **최소 1개 필수**.

#### 프론트엔드 화면

- **투자성향 설문/결과 화면**(`InvestmentProfileView`): 자산구간·위험·기간·목적은 라디오,
  관심 산업은 **칩 멀티선택**. 선택지는 `/options/`에서 동적으로 받아 렌더.
- 등록 여부에 따라 **결과 보기 → 수정 / 삭제** 흐름으로 전환.
- **마이페이지 '투자 성향' 탭**에 현재 성향 요약을 노출.
- `/investment-profile` 라우트는 `requiresAuth`(로그인 필요).

> 백엔드 테스트 13개 통과(등록·조회·수정·soft delete·검증·인증).

### 모델 설계

투자성향은 `accounts.User`와 **1:1(OneToOne)** 로 분리해, User 모델은 인증 책임만 갖도록 했다.

```python
class InvestmentProfile(models.Model):
    user = models.OneToOneField(settings.AUTH_USER_MODEL, on_delete=CASCADE,
                                related_name="investment_profile")  # 사용자당 1건
    available_asset   = ...  # 금액 구간 5단계 (1천만 미만 ~ 5억 이상)
    risk_type         = ...  # 안정형 / 안정추구형 / 균형형 / 적극투자형 / 공격투자형
    investment_period = ...  # 단타 / 스윙 / 중장기
    investment_goal   = ...  # 자산증식 / 안정적수익 / 배당수익 / 단기차익 / 노후대비
    sectors           = models.ManyToManyField(Sector)   # 관심 산업, 최소 1개
    is_active         = models.BooleanField(default=True) # F210 soft delete
```

**설계 의도**
- **관심 산업(`Sector`)을 enum이 아닌 별도 마스터 테이블 + M2M으로** 두었다. 추천 후보군(F300)·
  뉴스 키워드(F500)와 **같은 분류 체계를 공유**하기 위해서다. 시드 10종: 반도체·2차전지·방산·
  AI·에너지·게임·바이오·자동차·금융·인터넷·플랫폼.
- `accounts.User.has_investment_profile`은 **`is_active` 기준**으로 판단한다 — soft delete된
  프로필은 "미등록"으로 취급.
- enum 필드는 `*_display`로 **한글 라벨을 함께** 내려주고, `sectors`는 쓰기 시 산업 `code`
  목록을 받고 읽기 시 `code`+`name`(`sectors_detail`)으로 돌려준다.

### 어려웠던 점과 해결 방법

#### 1. 명세 해석 착오 — 점수 퀴즈로 잘못 구현
처음엔 투자성향을 **점수형 퀴즈**(설문 점수 → 유형 산출)로 구현했는데, 상세 기능명세서 §5.2는
**"선호 직접 입력 + CRUD"** 방식이었다.
→ 퀴즈용 점수/가중치 로직(`catalog.py`, `scoring.py`)을 걷어내고 직접 입력 CRUD로 **전면 교체**했다.
점수·가중치는 추천(F300 §5.3)의 책임으로 분리했다.

#### 2. 저장 도중 일부만 반영되는 위험
프로필 본문과 관심 산업(M2M)은 저장 시점이 달라, 중간에 실패하면 산업만 빠진 **반쪽 데이터**가
남을 수 있었다.
→ `update_or_create` + `sectors.set()`을 **`transaction.atomic`** 으로 묶어, 전부 성공이 아니면
전부 롤백되게 했다.

#### 3. 삭제 후 데이터 정합
하드 삭제하면 이후 F300 추천 이력/캐시와 엮였을 때 흔적이 사라진다.
→ **soft delete**(`is_active=False`)로 비활성화만 하고, 재등록은 같은 행을 upsert하도록 했다.

---

### 느낀점

- **명세를 추측으로 채우면 비싸다.** 점수 퀴즈로 한참 만든 걸 명세 한 줄 때문에 갈아엎었다.
  모호하면 코드보다 **명세 확인이 먼저**라는 걸 다시 배웠다.
- **한 기능의 데이터 모델이 다음 기능의 입력이다.** 관심 산업을 enum이 아닌 마스터 테이블로
  둔 선택이 F300/F500에서 분류 체계를 공유하게 해줄 것이다.
- **삭제는 "지우기"가 아니라 "상태 전이"** 로 보는 게 서비스에선 더 안전하다는 걸 체감했다.

## F400 종목 조회 기능 구현

**F400 종목 조회**는 국내 주식 종목을 **목록·검색·상세**로 보여주는 기능입니다. 현재가·등락률·
시가총액 같은 시세를 **토스증권 Open API**에서 받아 캐시하고, 외부 API 장애 시에도 화면이
깨지지 않도록 **폴백(지연표시)** 을 설계했습니다.

**기술 스택(추가)**
- 시세 데이터: **토스증권 Open API** (OAuth2 client_credentials)
- HTTP 클라이언트: `requests`
- 캐시: Django 캐시 (토큰·시세 TTL)

---

### 구현 내용

#### 백엔드 API (전부 인증 불필요, `AllowAny`)

| 기능 | Method | Endpoint |
|---|---|---|
| 종목 목록 (F401) / 검색 (F402) | GET | `/api/v1/stocks/?search=&market=&sector=&page=` |
| 필터용 섹터 목록 | GET | `/api/v1/stocks/sectors/` |
| 종목 상세 (F403~F407) | GET | `/api/v1/stocks/{code}/` |

- **목록·검색(F401·F402)**: 검색은 **종목코드 정확일치 우선 + 종목명 부분일치**.
  `market`/`sector` 필터와 페이지네이션(기본 20건, 최대 100건)을 지원한다. 비로그인도
  열람 가능(`AllowAny`).
- **상세(F403~F407)**: 현재가·전일종가·등락률·거래량·시가총액·기준시각을 제공한다.
  시세나 뉴스 일부가 실패해도 **페이지가 깨지지 않도록 부분 실패를 허용**한다.
- **시세 갱신 전략**: 페이지에 **보이는 종목만**, 그중 **TTL이 지난 것만** 외부 호출한다.
  실패하면 마지막 정상값으로 폴백하고 `is_delayed=true`로 표시(F404 지연표시).
- **첫 로드 가속**: 등락률용 전일종가(일봉)는 하루 한 번만 바뀌므로 종목당 **하루 1회**만
  받는다. `python manage.py sync_daily`로 아침에 유니버스 전체를 미리 캐시해 두면, 사용자의
  첫 목록 조회도 외부 호출 없이 **즉시 응답**(목록 ~수십 ms)한다. 토스 캔들은 단건 호출 +
  레이트리밋이 있어, 요청 경로에선 동시성을 낮게(`_CANDLE_WORKERS`) 두고 부족분은 sync
  재시도로 채운다.

#### 토스 Open API 연동

- OAuth2 토큰을 캐시·재사용하고, 만료(만료 60초 전 갱신)나 `401` 시 **1회 재발급 후 재시도**한다(F408).
- 자격증명 미설정 시 **네트워크 호출 없이 즉시 폴백**해 개발 환경에서 지연이 생기지 않게 했다.
- `get_prices`(현재가 다건)·`get_stocks`(기본정보)·`get_candles`(일봉)로 분리하고, 다건 조회는
  200개씩 청크로 나눠 호출한다.

#### 프론트엔드 화면

- **종목 목록 화면**(`StockListView`): **2단 마스터-디테일** 레이아웃.
  - 좌측: 검색바 + 스크롤되는 종목 목록
  - 우측: 인라인 상세 패널 — 현재가 박스, 시가총액(조/억), 산업군, **데이터 배지(실시간/기본데이터)**,
    뉴스 placeholder
- 행 클릭 시 우측 패널이 갱신되고, 진입 시 **첫 종목이 기본 선택**된다.
- `/stocks/:code?` 라우트(**공개**, 종목코드로 딥링크 가능).

> 백엔드 테스트 8개 통과(토스 클라이언트는 mock으로 대체). 토스 키가 없어도 가격을
> `null`로 두고 우아하게 동작.

### 모델 설계

종목 마스터(`Stock`)와 시세 캐시(`Quote`)를 분리했다.

```python
class Stock(models.Model):      # F400 유니버스 (시드 37종 국내 대형주)
    code, name
    market = "KOSPI" | "KOSDAQ"
    sector = models.ForeignKey("profiles.Sector", ...)  # F200과 같은 분류 체계 공유
    shares_outstanding = ...                            # 시총 계산용, 토스에서 채움

class Quote(models.Model):      # 시세 캐시 (Stock과 1:1)
    last_price, prev_close, volume, as_of
    updated_at                                          # TTL 판단 기준
    @property
    def change_rate(self):                              # (현재가 - 전일종가) / 전일종가
        ...
```

**설계 의도**
- 토스 Open API에는 **'종목 목록/검색' 엔드포인트가 없다**(심볼을 알아야 조회 가능). 그래서
  목록·검색·추천 후보군의 기준이 되는 **종목 집합(유니버스)을 `Stock` 테이블로 직접 보유**한다.
- 토스가 직접 주지 않는 값은 **우리가 계산**한다: 등락률 = **전일종가(일봉) 대비**,
  시가총액 = **발행주식수 × 현재가**.
- `Quote`에 **마지막 정상값**을 저장해 외부 장애 시 폴백(지연표시)으로 쓴다.
- `Stock.sector`를 F200의 `Sector`와 공유 → 이후 F300에서 **사용자 관심산업 매치율** 계산에 재사용.

### 어려웠던 점과 해결 방법

#### 1. '종목 목록' API가 없는 외부 데이터 소스
토스 Open API는 심볼을 알아야 조회할 수 있고, 종목 검색/목록 엔드포인트가 없었다.
→ 국내 대형주 **37종을 유니버스로 시드**해 `Stock` 테이블로 보유했다. 목록·검색·필터는 우리 DB
기준으로 처리하고, **시세만 외부 호출**한다.

#### 2. 외부 시세 API 장애가 화면을 깨는 문제
시세 호출이 느리거나 실패하면 목록·상세 전체가 멈출 수 있었다.
→ **TTL 캐시 + 마지막 정상값 폴백**으로 풀었다. 보이는 종목만, 신선하지 않은 것만 갱신하고,
실패하면 `is_delayed=true`로 "기본 데이터" 배지를 띄우되 **화면은 정상 렌더**한다.

#### 3. 토큰 만료 / 401 처리
client_credentials 토큰은 24시간 유효·리프레시가 없어서, 만료되면 호출이 `401`로 떨어졌다.
→ 토큰을 캐시(**만료 60초 전 갱신**)하고, `401`이면 **1회 강제 재발급 후 재시도**하도록 했다(F408).
자격증명이 없으면 호출 없이 즉시 폴백해 개발 환경 지연을 막았다.

#### 4. 토스가 주지 않는 파생 지표
등락률·시가총액을 토스가 직접 주지 않았다.
→ **일봉 2개**로 전일종가를 얻어 등락률을, `get_stocks`의 **발행주식수 × 현재가**로 시가총액을
직접 계산했다. 일봉·발행주식수는 자주 변하지 않으므로 **일 단위로만 갱신**(`daily_as_of`)한다.

#### 5. 등락률이 종목을 클릭해야만 보이던 문제 (+ 첫 로드 속도)
처음엔 전일종가(`prev_close`)를 **상세 조회에서만** 채워서, 목록에선 종목을 일일이 클릭해
상세를 한 번 연 종목만 등락률이 보였다.
→ 목록 뷰에서도 `ensure_daily_candles(page)`로 **전일종가를 미리 채워**, 클릭 없이 등락률이
바로 뜨게 했다.

그런데 토스 캔들은 **종목당 단건 호출**이라 목록 첫 로드가 느렸다. 그래서:
- 첫 종목을 **동기로 호출해 토큰을 데운 뒤** 나머지를 병렬화(토큰 동시 재발급 herd 방지).
- 동시성을 **높였더니 오히려 레이트리밋으로 성공률이 떨어졌다**(8워커 22/37 → 16워커 10/37 성공).
  → 동시성을 **낮게**(`_CANDLE_WORKERS=4`) 두는 게 더 빠르고 안정적이었다.
- 토스가 데이터를 안 주는 종목까지 **매 요청 재호출**하던 걸, 빈 응답도 "오늘 시도함"으로 기록해 차단
  (단 전송 실패 `TossError`는 기록하지 않아 다음에 재시도).

근본 해결은 **요청 경로에서 캔들을 부르지 않는 것**이었다. 전일종가는 하루 한 번만 바뀌므로,
`python manage.py sync_daily`(저동시성 + 재시도 패스)로 **아침에 유니버스 전체를 미리 캐시**한다.
실측: 프리워밍 후 목록 첫 응답이 **약 2.7초 → 27ms**로, 이후 호출은 **~4ms**로 떨어졌다(37/37 등락률 채움).

---

### 느낀점

- **외부 API는 "되는 경우"보다 "안 되는 경우" 설계가 본론**이었다. 폴백·TTL·401 재시도 같은
  실패 경로가 기능의 절반을 차지했다.
- **내가 가진 데이터(유니버스)와 외부에서 받는 데이터(시세)를 나누니** 책임이 또렷해졌다.
  외부에 없는 건 직접 보유하거나 계산하면 된다.
- **데이터가 그럴듯한지 의심하는 것도 검증의 일부**다. 토스 시세가 실제와 달라 보여
  (샌드박스/테스트값 의심) 키 환경(prod 여부) 확인이 필요하다고 기록해 두었다.
- **"빠르게"는 더 많이 부르는 게 아니라 안 부르는 것**이었다. 동시성을 올릴수록 레이트리밋으로
  오히려 느려졌고, 결국 자주 안 바뀌는 데이터(전일종가)를 요청 경로 밖에서 미리 캐시(`sync_daily`)하는
  게 정답이었다.

## F300 AI 종목 추천 기능 구현

**F300 AI 종목 추천**은 F200에서 저장한 투자 성향과 F400 종목 유니버스를 바탕으로
사용자별 추천 종목을 산정하고, 추천 이유를 자연어로 생성하는 기능입니다. 추천 점수는
관심 산업·위험성향 적합도·기간수익률·유동성을 조합해 계산하고, 추천 이유는 SSAFY GMS API를
우선 사용하되 실패하면 규칙 기반 템플릿으로 폴백하도록 설계했습니다.

**핵심 결정**: 추천 이유는 단순 문장 생성이 아니라 **추천 선정 결과 + 점수 근거 + 뉴스 요약**을
GMS 프롬프트에 함께 넣어 생성한다. 사용자는 추천 카드에서 실제 프롬프트 근거로 사용된
뉴스 기사 링크를 펼쳐 볼 수 있다.

**기술 스택(추가)**
- AI 문장 생성: **SSAFY GMS API** (OpenAI Responses API 프록시)
- 뉴스 요약: Google News RSS 검색 결과 기반 헤드라인 요약
- HTTP 클라이언트: `requests`
- 배치/점검 명령: `compute_metrics`, `sync_news_summaries`, `probe_gms`

### 구현 내용

#### 백엔드 API (전부 인증 필요)

| 기능 | Method | Endpoint |
|---|---|---|
| 활성 추천 조회 | GET | `/api/v1/recommendations/` |
| 추천 생성·재생성 | POST | `/api/v1/recommendations/` |

- **추천 후보 구성**: 활성 종목(`Stock.is_active=True`) 전체를 후보군으로 삼고, F200 투자성향과
  종목의 산업 분류를 함께 사용한다. 사용자의 활성 투자성향이 없으면 `400 PROFILE_REQUIRED`를 반환한다.
- **점수 산정**: `sector_match`, `risk_fit`, `momentum`, `liquidity` 네 요소를 0~1로 계산하고,
  투자목적·투자기간에 따라 가중치를 조정해 0~100점 점수로 변환한다.
- **시세 파생 지표**: 토스 일봉 데이터를 기반으로 변동성·모멘텀·유동성을 `StockMetric`에
  스냅샷으로 저장한다. 추천 요청 때마다 장기 일봉을 다시 부르지 않도록 `compute_metrics` 명령으로
  미리 계산할 수 있게 했다.
- **뉴스 요약**: 추천 상위 종목에 대해 RSS 뉴스 요약(`StockNewsSummary`)을 TTL 기준으로 갱신한다.
  네트워크 실패나 RSS 오류가 있어도 추천 생성은 계속 진행된다.
- **GMS 추천 이유 생성**: GMS 프롬프트에는 추천 순위, 최종 점수, 점수 구성, 시세 지표,
  최근 뉴스 요약을 함께 넣는다. GMS 키가 없거나 호출이 실패하면 해당 종목만 규칙 기반 문장으로 폴백한다.
- **추천 결과 영속화**: 새 추천을 만들 때 기존 활성 추천은 `is_active=False`로 무효화하고,
  새 `Recommendation`과 `RecommendationItem`을 저장한다. 추천 이유 생성에 사용된 뉴스 기사 링크는
  `RecommendationItem.news_articles`에 스냅샷으로 남긴다.

#### GMS / 뉴스 연동

- GMS 호출 엔드포인트는 `https://gms.ssafy.io/gmsapi/api.openai.com/v1/responses` 형식을 사용한다.
  `GMS_API_KEY`만 `.env`에 넣으면 기본 base/model(`gpt-4.1`)로 호출된다.
- `python manage.py probe_gms`로 추천 생성 전체를 거치지 않고 GMS 연결만 단독 확인할 수 있다.
- `python manage.py sync_news_summaries --code 005930`처럼 특정 종목의 뉴스 요약만 미리 갱신할 수 있다.
- 뉴스 기사 링크는 추천 생성 시점의 스냅샷으로 저장한다. 이후 RSS 결과가 바뀌어도 사용자가 보는
  기사 목록은 해당 추천 이유를 만들 때 사용한 근거와 일치한다.

#### 프론트엔드 화면

- **맞춤 종목 추천 화면**(`RecommendView`): 추천이 없으면 생성 CTA를 보여주고, 추천이 있으면
  순위·종목명·현재가·등락률·추천 이유·추천 점수를 리스트로 보여준다.
- `reason_source`에 따라 추천 이유가 **AI 생성(GMS)** 인지 **규칙 기반**인지 표시한다.
- 추천 카드를 클릭하거나 `뉴스 보기` 버튼을 누르면 카드가 펼쳐지고, 추천 이유 생성에 사용된
  뉴스 기사 링크들이 표시된다. 링크는 새 탭으로 열리도록 했다.
- `/recommendations` 라우트는 `requiresAuth`(로그인 필요).

> 백엔드 전체 테스트 38개 통과. 추천 앱 테스트는 GMS 호출·뉴스 RSS 요약·프롬프트 내용·추천 API 응답을
> mock 기반으로 검증한다. 프론트엔드 `npm.cmd run build`도 통과.

### 모델 설계

추천 결과는 **현재 활성 추천 1건 + 추천 항목 N건 + 지표/뉴스 스냅샷**으로 나눴다.

```python
class StockMetric(models.Model):
    stock = models.OneToOneField(Stock, related_name="metric")
    volatility = ...  # 일간수익률 표준편차
    momentum = ...    # 기간수익률
    liquidity = ...   # 평균 일거래대금
    as_of = ...       # 지표 스냅샷 일자

class StockNewsSummary(models.Model):
    stock = models.OneToOneField(Stock, related_name="news_summary")
    summary = ...     # GMS 프롬프트에 넣을 뉴스 요약
    articles = ...    # title/url/source/published_at 목록
    updated_at = ...  # TTL 판단 기준

class Recommendation(models.Model):
    user = models.ForeignKey(User, related_name="recommendations")
    is_active = True
    risk_type = ...       # 생성 시점 성향 스냅샷
    reason_source = ...   # template | gms

class RecommendationItem(models.Model):
    recommendation = models.ForeignKey(Recommendation, related_name="items")
    stock = models.ForeignKey(Stock)
    rank, score, reason
    breakdown = ...       # 점수 구성요소
    news_articles = ...   # 추천 이유 생성에 사용된 기사 링크 스냅샷
```

**설계 의도**
- 추천 점수와 추천 이유는 생성 시점의 상태를 보존해야 하므로 `RecommendationItem`에
  `score`, `breakdown`, `reason`, `news_articles`를 저장한다.
- 시세 파생 지표(`StockMetric`)는 추천 요청마다 외부 API를 다시 부르지 않도록 별도 스냅샷으로 분리했다.
- 뉴스 요약(`StockNewsSummary`)은 종목별 최신 캐시이고, 추천 항목의 `news_articles`는
  추천 결과에 귀속되는 스냅샷이다. 이 둘을 나눠야 "현재 최신 뉴스"와 "추천 당시 근거"가 섞이지 않는다.
- GMS는 외부 의존성이므로 `GmsClient`를 provider로 감싸 테스트에서 fake 구현을 주입할 수 있게 했다.

### 어려웠던 점과 해결 방법

#### 1. 추천 이유가 단순 템플릿처럼 보이는 문제
처음엔 GMS 프롬프트에 종목·성향·모멘텀 정도만 들어가서, 사용자가 왜 이 종목이 선정됐는지
충분히 납득하기 어려웠다.
→ 프롬프트에 **추천 순위·최종 점수·점수 구성·시세 지표·뉴스 요약**을 모두 넣어,
추천 결과 자체를 설명 근거로 사용하도록 바꿨다.

#### 2. GMS 호출이 실패해도 화면이 조용히 규칙 기반으로 보이는 문제
외부 호출 실패 시 폴백은 필요하지만, 개발 중에는 왜 AI 생성이 안 되는지 알기 어려웠다.
→ `probe_gms` 명령으로 GMS 단독 연결을 검증할 수 있게 하고, GMS 실패 시 서버 로그에
종목코드와 실패 사유를 남기도록 했다.

#### 3. 뉴스 요약과 기사 링크의 정합성
처음에는 뉴스 요약 텍스트만 GMS에 전달하면 된다고 생각했지만, 사용자가 추천 근거를 확인하려면
요약에 사용된 실제 기사 URL도 필요했다. 또한 나중에 RSS가 바뀌면 추천 이유와 기사 목록이 어긋날 수 있었다.
→ RSS에서 `title/url/source/published_at`을 함께 저장하고, 추천 생성 시점의 기사 목록을
`RecommendationItem.news_articles`에 **스냅샷**으로 복사했다.

#### 4. 외부 뉴스 수집 실패가 추천 전체를 막는 위험
뉴스 RSS는 키 없이 쓸 수 있지만 네트워크·검색 결과·RSS 파싱 상태에 영향을 받는다.
→ 뉴스 수집은 best-effort로 처리했다. 실패하면 로그만 남기고, GMS 프롬프트에는
`수집된 뉴스 요약 없음`을 넣어 추천 생성 자체는 계속 진행한다.

#### 5. 기존 추천 결과가 계속 규칙 기반으로 보이던 문제
`.env`에 GMS 키를 넣어도 이미 저장된 활성 추천은 자동으로 다시 생성되지 않는다.
GET은 기존 결과를 보여주고, POST를 해야 새 추천 이유가 생성된다.
→ 추천 화면에 `다시 추천받기` 흐름을 두고, 개발 중에는 실제 사용자 추천을 재생성해
`reason_source='gms'`와 뉴스 기사 스냅샷이 저장되는지 확인했다.

---

### 느낀점

- **추천 기능은 "점수 계산"보다 "설명 가능성"이 더 중요했다.** 점수만 높다고 사용자가 납득하는 게 아니라,
  왜 선정됐고 어떤 뉴스 맥락이 있었는지 함께 보여줘야 추천처럼 느껴진다.
- **AI 호출은 마지막 문장 생성 단계일 뿐, 입력 데이터 설계가 더 큰 비중을 차지한다.**
  GMS가 좋은 문장을 만들려면 프롬프트에 들어가는 점수·뉴스·성향 정보가 먼저 정리돼 있어야 했다.
- **외부 의존성은 항상 폴백과 관측 수단이 필요하다.** GMS, RSS, 토스 API 모두 실패할 수 있으므로
  기능이 멈추지 않는 폴백과 실패를 확인할 수 있는 로그/명령이 같이 있어야 했다.
- **스냅샷을 남기는 기준이 중요했다.** 추천 당시 근거와 현재 최신 데이터는 다르기 때문에,
  추천 결과에 귀속되는 정보는 따로 저장해야 나중에 설명이 흔들리지 않는다.
